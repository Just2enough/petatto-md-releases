# Petatto.md が「やらないこと」（Non-Goals）

> Petatto.md が **意図的に実装しない / 当面対象としない** 範囲をまとめる。これは機能不足ではなく、「Obsidian で編集し、デスクトップでは眺める・チェックする」というコアコンセプトを保つための明確なスコープ境界。

## 前提：Petatto.md は Obsidian の補助

Petatto.md は **Obsidian を置き換える md エディタではない**。編集の主役は Obsidian、デスクトップ側は付箋として「眺める・軽くチェックする」場所。この前提から外れる機能は基本的に non-goal とする。

## レンダリング範囲

- 画像の表示
- コードブロックのレンダリング（シンタックスハイライト含む）
- テーブル・引用・水平線
- wiki リンク `[[...]]` のクリック遷移、および http/https 以外のスキーム（`mailto:` / `file:` / `obsidian://` 等）のリンク遷移
- 見出し内の強調記法

> サポートするのは見出し・段落・リスト・チェックボックス・強調（`**太字**` / `*斜体*` / `~~取消~~`）・http/https リンクのクリック遷移まで。サポート外の要素はプレーンテキストとして表示する。リッチな閲覧は Obsidian 側で行う前提。frontmatterも表示しない。

## 付箋のカスタマイズ範囲

- ユーザ定義パレットの作成・編集 UI（ビルトイン 2 種「ミルキー / ビビッド」× 各 4 色のラジオ切替のみ）
- 単色パレット（slots=1）/ 5 色以上（N>4）のスロット
- 色を frontmatter で指定する機能（例: `petatto-md-color: pink`）
- フォント名・行間のカスタマイズ（Meiryo / `line-height: 1.6` 固定）

> 本文フォントサイズのみ 5 段階（11/12/13/15/17px）の選択 UI を提供する。DB スキーマ・`Palette` 型・`list_palettes` API は将来のユーザ定義パレットに耐える構造を確保済みだが、編集 UI は当面持たない。

## vault / ファイル管理

- 複数 vault の同時利用（単一 vault のみ）
- 外部（Obsidian / エクスプローラ等）でのリネーム・移動への追従（`<旧パス削除> + <新パス追加>` として個別処理する）
- 付箋同士のグループ化・タグ付け
- シンボリックリンク（symlink）の追従。実体ファイルへの symlink になっている `.md` は付箋として認識しない（vault スキャンは通常ファイルのみを対象とし、リンクは辿らない）

> アプリ主導のファイル名変更（付箋ヘッダからのリネーム）は対応する。アプリ起動中の「行方不明（外部削除・移動）」の検知と編集内容の退避も対応する。対象外なのは「外部リネームの自動追従」。

> **シンボリックリンクについて**: Petatto.md の vault スキャンは通常ファイルのみを対象とし、symlink は辿らない。Obsidian でノートを別の場所と共有するために symlink を使っている場合、その実体への symlink な `.md` は付箋化できない（Windows ではファイル symlink の作成に管理者権限 / 開発者モードが必要なため、通常運用では遭遇しない想定）。なお、vault 内に外部を指す symlink が置かれた場合の **セキュリティ上の封じ込め**（実体ファイルを vault 外へ脱出させない防御）は別途実装済みで、本項はあくまで「symlink な md を付箋として扱わない」という機能スコープの話。

## プラットフォーム / 配布

- mac / Linux 対応（Windows 専用）
- 複数 PC 間の同期（位置共有など）
- コード署名（未署名 .msi 配布。初回の SmartScreen 警告は許容）
- テレメトリ収集・クラッシュレポートの自動送信（行わない。ログはローカルのみ保存し、不具合時はユーザーが手動でログを提出する）

## メインウィンドウ / 編集体験

- メインウィンドウのダッシュボード化 / 付箋一覧テーブル / 個別操作 UI（メインは設定 + 状態サマリに徹する。個別操作はトレイメニューと付箋ヘッダの UI から行う）
- メインウィンドウのサイドバーナビゲーション / タブ切替
- 編集の別ウィンドウ化（同一ウィンドウ内 textarea 方式。v1.1.0 以降で再検討）
- 編集中の競合 3-way マージ画面（「上書き / 破棄して再読込」の 2 択ダイアログのみ）
- 編集モードの Ctrl+Enter ショートカット / ダブルクリック位置への厳密なキャレット配置

## i18n の現状スコープ

UI 文字列は両側（フロント自前 + Rust の rust-i18n）でプラミング済みだが、現状の取り扱いには以下の境界がある。

- **英訳テキストは未投入**: `src/i18n/locales/en/*.json` と `src-tauri/locales/en.json` は同一キーで値が空の雛形のみ。未投入キーは ja にフォールバックする
- **言語切替 UI は当面なし**: 実稼働は ja 固定（フロント `src/i18n/index.ts` の `locale` 定数 / Rust `lib.rs` の `set_locale("ja")` で集約）。OS ロケール検出・設定 UI からの切替は持たない
- **Rust 側 UI 文字列は rust-i18n 管轄**: フロント自前 i18n（`t()`）は Rust プロセスが生成する文字列に届かない。トレイメニュー / 起動通知 / 表示エラー reason などは Rust 側で `t!` 経由でローカライズする。

以下は i18n 対象外（据え置き）:

- コード内コメント、`console.*` / `log::*` 等のログ文言（UI ではない）
- `db.rs` 等の例外時のみ表示される裏方エラー（DB lock 等）
- `vault.rs` / `frontmatter.rs` の稀な IO 失敗

> 英訳投入と言語切替 UI は 将来の検討余地として残すが、現 MVP は ja 固定運用を前提とする。
